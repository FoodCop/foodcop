## Recommendation Engine (finally has real inputs)

`public.recipes.flavor_profile` and `public.profiles.food_dna` (jsonb columns) plus a working SQL function `public.get_recommended_recipes(p_user_id, p_limit, p_offset)` (real euclidean-distance flavor matching, random fallback when `food_dna` is empty) have existed since the very first migration — but **nothing ever wrote to either column**. `scripts/seed_recipes.ts` always uploaded `r.flavor_profile || {}`, and `curatedRecipes.json` (1,251 Spoonacular-derived recipes) never had real flavor-axis data to begin with, since Spoonacular doesn't provide it.

This session closed that gap instead of duplicating the RPC's matching logic in TypeScript:

- **`src/lib/services/recommendationService.ts`** → `aggregateForUser(userId)` computes a recency-weighted (90-day half-life decay) average `FlavorVector` from the user's own `food_cards`, plus cuisine/diet tag-frequency counts, and **writes the vector to `profiles.food_dna`** — the first real data that column has ever held. `getRecommendedRecipes()` then calls the existing `get_recommended_recipes` RPC directly once a user has ≥1 food card ("warm" path); for a brand-new user with zero cards ("cold start") it instead ranks `curatedRecipes.json` by cuisine/diet overlap against their onboarding `taste_profiles.cuisines`/`dietary` — real signal, not a fabricated flavor match. `getNearbyRestaurants()`/`getSuggestedVideos()` feed the dashboard's other two rails off the same aggregate's top cuisine.
- **`scripts/enrichRecipeFlavorProfiles.ts`** (new) — the piece that makes the *warm* path meaningful for the curated dataset: one Gemini call per recipe (title/cuisines/dishTypes/diets/ingredients as context) asking only for the 10 flavor axes, checkpointed back into both `curatedRecipes.json` and `public.recipes.flavor_profile`. **Now complete — verified, not just reported: all 1,251/1,251 recipes have a populated, non-zero `flavorProfile` in `curatedRecipes.json`, and all 1,251/1,251 rows in the live `public.recipes` table have a populated `flavor_profile` (checked directly against the local Docker Postgres via `docker exec ... psql`, not just the JSON checkpoint).** Concurrency was turned down to 1 and retries up to 10 mid-run (from the script's original 5/3) to work around Gemini rate-limiting on the full run. This means `get_recommended_recipes`'s "warm path" now has real flavor vectors on both sides of the match for every curated recipe, not just for users with food_cards — the recommendation engine is fully real end-to-end for the first time.
- No new tables were needed for any of this — `profiles.food_dna` and `recipes.flavor_profile` already existed and were simply never populated.
- **Deliberately not touched:** `dna.ts`'s 25-question 5-module quiz (adventure/luxury/comfort/social/health scores) — fully-built scoring logic, but no live route/UI ever calls it and `taste_profiles.dna_scores` is never written. ## Styling architecture (the big consolidation)

One master stylesheet: `src/scss/main.scss` → `_variables.scss` (FUZO brand colors: chili/turmeric/lime/sky/rose as Bootstrap `$primary`/etc. overrides) → real `bootstrap/scss/bootstrap` → feature partials (`_header.scss`, `_scout.scss`, `_auth.scss`, `_profile.scss`, `_dna.scss`, `_feed.scss`, `_trims.scss`, `_rewards.scss`, `_tako.scss`, `_onboarding.scss`, `_bites.scss`, `_dashboard.scss`, `_studio.scss`), imported once in the root layout. (Create Card briefly regressed this rule with raw Tailwind classNames — fixed same session, see Gotcha #13.) **Do not** import a template's own compiled CSS bundle again — hand-port the specific rules a component needs into a partial instead. This replaced an earlier (bad) approach of importing each template's whole CSS file per route, which caused width inconsistencies and eventually broke the Profile page outright.

**This app has no Tailwind, on purpose.** Ported components sometimes carry over Tailwind className strings from the legacy React app (which used Tailwind) — those are dead weight here and render unstyled. Don't install Tailwind to fix it: Bootstrap and Tailwind both ship utility classes with identical names but different values (`.mt-3`, `.p-4`, `.gap-2`, etc.), so adding Tailwind risks silently corrupting spacing across the *entire* app depending on CSS load order. Hand-port Tailwind classNames to a feature SCSS partial instead (see `_scout.scss`/`_rewards.scss`/`_tako.scss` for the pattern).

**Icon strategy:** `FUZO_V3/public/SVG/` has an 837-file icon library organized by category (`restaurant/`, `food/`, `foodies/`, `social/`, `map/`, `UI/`, `flags/`, ...). Don't bulk-copy it — copy only the specific icons a feature actually references into `fuzo-next/public/SVG/`, preserving the folder structure (Rewards copied 30, Tako copied 8 more, Onboarding's card-picker/result icons this session). These SVGs have **hardcoded fill colors baked in, not `currentColor`** — CSS `color`/`fill` inheritance tricks don't tint them. To render one as a clean white silhouette on a colored badge, use `filter: brightness(0) invert(1)` (see Tako's Quick Discovery/Quick Links cards); otherwise they just render in their own native colors, which is fine for badge/mood icons sitting on a neutral surface.

**For generic UI icons (search, clock, close, filter sliders, chevrons, tabs)** — as opposed to food/cuisine/badge imagery — use `lucide-react`. It's a real, already-used dependency (Chat, Scout, Rewards, Tako, and now Bites all import from it directly), not something to add. The `/SVG/` library doesn't have this kind of generic icon anyway (it's all food/restaurant/social-themed).

## Known gotchas (read before touching build config again)

1. **Turbopack can't compile our Bootstrap SCSS on this machine** — `@import "mixins/banner"` inside `node_modules/bootstrap/scss/bootstrap.scss` fails to resolve under Next 16's default Turbopack dev/build, but works fine under webpack. That's why `package.json` scripts force `--webpack`. Don't remove that flag without re-testing.
2. **`wowjs` was removed entirely.** It's a legacy UMD package with fragile bundler interop (broke differently under Turbopack vs webpack), and its required `animate.css` companion was never even wired in, so it wasn't producing any visible effect anyway.
3. **Bootstrap's JS must be loaded as a plain `<script>` tag, not a JS import.** A dynamic `import()` silently failed to build as a webpack chunk (`ChunkLoadError` timeout); a static import broke SSR (`document is not defined`, since Bootstrap's UMD touches `document` at eval time). Fix: `bootstrap.bundle.min.js` is copied into `public/vendor/` and loaded via `<Script src="/vendor/bootstrap.bundle.min.js" strategy="afterInteractive">` in the root layout — exactly how the original templates loaded it.
4. **Never use Bootstrap's JS-driven offcanvas for anything with a nav Link inside it.** Bootstrap's offcanvas appends a full-viewport backdrop div directly to `<body>`, outside React's tree. If a link inside it triggers Next.js client-side navigation while the close transition is still running, the backdrop can be orphaned — invisible, but it permanently blocks every click on every subsequent page (no console error, easy to mistake for "navigation is broken"). Fix applied: `SiteHeader`'s drawer + dropdowns are now 100% React `useState`-controlled, no `data-bs-toggle`/`data-bs-dismiss` at all. If you add another offcanvas/modal with links inside, do the same.
5. **`SiteHeader` must live in a shared layout, not be rendered per-page.** Re-mounting it on every navigation was part of the above race condition. `(main)/layout.tsx` renders it once. `TakoWidget` follows the same rule for the same reason.
6. **K: drive (exFAT) cannot run Next.js dev/build reliably** — concurrent directory operations Next.js relies on fail with EPERM/access-denied on that filesystem. Everything happens on C: now.
7. **`.env.local` has `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY` reused from the old app** for now — fine for dev, rotate before anything resembling production.
8. **Flexbox/Grid mobile overflow footgun:** a flex item defaults to `min-width: auto`, so a horizontally-scrolling row nested inside a flex column (chips, pills, card carousels) can force its ancestor wider than the viewport instead of scrolling internally — and if an ancestor has `overflow: hidden` (common for full-bleed shells), the excess just gets silently clipped rather than showing a scrollbar. Fix: `min-width: 0` on the ancestor flex item. Same root cause with CSS Grid: bare `1fr` tracks size to content's min-content width, so long unbroken labels can blow out a column — use `minmax(0, 1fr)` instead. Hit and fixed on Tako; **not yet verified** on Rewards' achievement grid or Scout's wizard cuisine/photo grids, which use the same bare-`1fr` pattern.
9. **Prefer wrapping (`flex-wrap: wrap`) over horizontal scroll (`overflow-x: auto`) for chip/pill rows where content length is unbounded** — sidesteps gotcha #8 entirely instead of needing the `min-width: 0` fix. Used for Tako's Food Mood pills after the overflow bug above.
10. **A file existing under `src/` (even one that type-checks cleanly) doesn't mean it's a real implementation — check every "stub"-named file individually.** The pre-existing `src/components/bites/` port had both `feedStub.ts` (a genuine no-op — every method just returns success without doing anything) and `supabaseStub.ts` (misleadingly named — it actually re-exported the real, working Supabase client). Don't assume based on a filename; read each one. Also don't trust a clean `tsc --noEmit` as proof a ported feature works — the old Bites port type-checked perfectly while being 100% unstyled (see Tailwind gotcha above) and backed by fake Unsplash fallback data instead of this app's real `curatedRecipes.json`.

18. **A PL/pgSQL function's `RETURNS TABLE (col_name ...)` auto-declares `col_name` as a local variable inside the function body — if that name matches a real column on a table the function also `UPDATE`s, every unqualified reference to that column inside the function becomes ambiguous.** Hit in `award_points()` (`supabase/migrations/20260719000000_points_and_gamification.sql`): the return signature originally named an output column `points_total`, and the function's own `UPDATE public.users SET points_total = points_total + v_points ...` immediately failed with `column reference "points_total" is ambiguous` the first time it actually ran (a `CREATE FUNCTION` with no syntax error gives zero warning about this — it only surfaces at call time). Fixed by prefixing every `RETURNS TABLE` output column `out_*`, leaving the function body free to reference the real table columns unqualified.
